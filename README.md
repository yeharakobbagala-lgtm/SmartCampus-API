📌 Project Overview
SmartCampus is a professional-grade RESTful API developed using Java EE (JAX-RS). The system is designed to manage university infrastructure by tracking Rooms, Sensors (CO2, Temperature), and real-time Readings.

The project demonstrates advanced backend concepts including HATEOAS, sub-resource delegation, global exception mapping, and observability through logging filters.

🚀 Key Features by Phase
Phase 1: Discovery & Hypermedia
HATEOAS Implementation: A Discovery Endpoint provides entry points to the API via dynamic JSON links.

Standardized Responses: Uses consistent JSON structures for API metadata.

Phase 2: Room Management
CRUD Operations: Full support for Creating, Reading, Updating, and Deleting rooms.

In-Memory Storage: Uses a thread-safe Mock Database with a Singleton DAO pattern.

Idempotency: Implements idempotent DELETE operations for reliability.

Phase 3: Sensor Operations & Search
Linking: Sensors are logically linked to Rooms via ID references.

Search & Filtering: Supports searching sensors by type (e.g., ?type=CO2) using JAX-RS Query Parameters.

Integrity Checks: Validates that a Room exists before allowing a Sensor to be added (422 error handling).

Phase 4: Sub-Resources (Readings)
Deep Nesting: Implements the Sub-Resource Locator pattern for paths like /sensors/{id}/readings.

State Management: Adding a new reading automatically updates the lastReading value on the parent Sensor.

Phase 5: Advanced Error Handling & Logging
Leak-Proof API: Custom Exception Mappers prevent Java stack traces from reaching the user.

Observability: A Global Logging Filter records every incoming request (Method/URI) and outgoing response (Status Code) in the server logs.

Semantic Errors: Returns accurate status codes like 409 Conflict (for busy rooms) and 422 Unprocessable Entity.

Question: In your report, explain the default lifecycle of a JAX-RS Resource class. Is a new instance instantiated for every incoming request, or does the runtime treat it as a singleton? Elaborate on how this architectural decision impacts the way you manage and synchronize your in-memory data structures (maps/lists) to prevent data loss or race conditions.
Answer: 
Lifecycle and Data Management by default JAX-RS uses a per request lifecycle. This means the system creates an instance of the resource class for every single HTTP request. Then it gets rid of this instance once the response is sent. So, any data that is stored in the usual instance variables is lost when we make a call, to the server.
Architectural Impact to make sure the data stays around we keep the API logic and the data storage separate. We do this by using something called Singleton Data Access Objects or DAOs for short and collections that store data in the server’s memory. These collections stay around for long as the server is running. This way JAX-RS Data Management is handled properly. The data stays safe.
When we talk about Concurrency and Synchronization, we have to think about how JAX-RS works. It is multi-threaded. This means many users can use these collections at the same time. This can cause problems. We call these problems race conditions. To stop our data from getting damaged or to prevent IDs we use special lists. These lists are thread-safe. We use things, like synchronized blocks or CopyOnWriteArrayList to manage our lists. This makes sure that one request can change the database at one time. This happens fast in just one millisecond. This keeps our data safe and accurate across the SmartCampus system. We do this to keep the SmartCampus system working correctly.
1.2 The Discovery Endpoint
Question: Why is the provision of Hypermedia (links and navigation within responses) considering a hallmark of advanced RESTful design (HATEOAS)? How does this approach benefit client developers compared to static documentation?
Answer: 
HATEOAS and Advanced RESTful Design
While it makes an API self-descriptive, HATEOAS (Hypermedia as the Engine of Application State) is a defining feature of sophisticated REST design. The server includes "navigation links" in the JSON response, saving the client from having to hardcode or memorize each URL. Similar to a website that only displays a "Next" button once a form has been completed, this informs the client exactly what they may do next.
Benefits to Client Developers 
This approach is much better than static documentation because it reduces coupling. If the server team changes the URL path, the client application won't break because it is programmed to follow the links provided by the server rather than using a hard-coded address. It makes the API easier to use and explore, as the responses themselves act as "live instructions" that are always up to date, unlike a PDF manual which can quickly become old.

Question: Is the DELETE operation idempotent in your implementation? Provide a detailed justification by describing what happens if a client mistakenly sends the exact same DELETE request for a room multiple time.
Answer: The DELETE operation is indeed idempotent. This implies that the database's final state doesn't change if a client sends the exact same request more than once. The room is located and deleted from the Mock Database upon the initial request, producing a 204 No Content success response. My code will search for that room, determine it is already gone, and return a 404 Not Found if the client submits the same request again. The procedure is still regarded as idempotent even when the response code changes from 201 to 404 since the room stays destroyed despite the second and third queries not changing the database.

3.1 Sensor Resource & Integrity
Question: We explicitly use the `@Consumes(MediaType.APPLICATION_JSON)` annotation on the POST method. Explain the technical consequences if a client attempts to send data in a different format, such as `text/plain` or `application/xml`. How does JAX-RS handle this mismatch?
Answer: When use the @Consumes(MediaType.APPLICATION_JSON) annotation I am telling my API to accept data that is formatted as JSON. If a client tries to send data in a format like text/plain or application/xml, the JAX-RS runtime will stop the request before it even reaches my code.
The JAX-RS runtime handles this situation by returning an HTTP 415 Media Type error to the client. This is an important feature for security and stability of the Java application because it prevents the server from trying to process bad data that could cause the Java application, to crash.
The @Consumes(MediaType.APPLICATION_JSON) annotation is crucial because it helps my API to work with JSON data and prevents problems that could happen if the server tries to process data that is not formatted as JSON.

Part 5: Advanced Error Handling, Exception Mapping & Logging
5.1	 Dependency Validation (422 Unprocessable Entity)
Question: Why is HTTP 422 often considered more semantically accurate than a standard 404 when the issue is a missing reference inside a valid JSON payload?
Answer: HTTP 422 (Unprocessable Entity) is more semantically accurate than a 404 because of the nature of the error. A 404 Not Found typically indicates that the server is unable to locate the page at all and that the "address" or URL you entered is incorrect. The URL is right when a client delivers a valid JSON payload to build a sensor, but the data such as a room Id that doesn't exist is illogical. By using 422, the developer is informed: "I received your data and I understand what you want, but I cannot process it because the room link you provided is impossible." Compared to a general 404, this particular feedback greatly simplifies debugging for client-side developers.
